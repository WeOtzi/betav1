### Changed
- **Estados de cotizacion alineados con finalizacion verificable (2026-06-09)**: el cambio valido para trabajos activos pasa por `in_progress -> artist_completed -> completed`; el estado `completed` queda reservado para la aceptacion final del cliente y requiere que no haya disputas abiertas. Se mantiene `in_progress -> pending` para el envio final de borradores de cotizacion y se actualizaron labels, conteos, archivo, calendario, administracion y tests para reconocer `artist_completed`.
- **Registro de artista sin paso de ciudad independiente (2026-06-01)**: `/register-artist` queda en 11 pasos y elimina el antiguo paso 04 de ciudad. La ciudad/ubicacion del artista ahora se deriva de la direccion de trabajo seleccionada: artistas de estudio o mixtos guardan la ciudad de la sede principal/seleccionada, y artistas independientes guardan la ciudad de su propia direccion. Se actualizo el guardado de drafts/finalizacion, el resumen de revision y las pruebas para cubrir que no exista `#city` ni `data-step="12"` y que `artists_db.city`/`ubicacion` salgan de la sede. Verificacion: `node --test tests/*.test.js` (91/91), chequeo DOM local del wizard (`IG / 11`) y consultas Supabase de columnas/datos de `artists_db` + `studio_locations`.
- **QA integral artista/cliente cerrada (2026-05-27)**: se probo el app local en `http://127.0.0.1:4560` de principio a fin para flujos de artista y cliente: registro closed beta + finalize/auth, cotizacion directa, mis cotizaciones, calendario, estadisticas, archivo, visitantes, marketplace, perfil publico, job board request, postulacion de artista, accept application y cotizacion desde job board. - **Version 2.0.1 (2026-05-27)**: se alineo la version del workspace en `package.json`, `package-lock.json`, `public/shared/js/app-config.json` y `README.md` para registrar el hardening responsive del registro de artista como patch release sobre la version publica `2.0.0`.
- **Dashboard de artista valida perfil completo antes de renderizar (2026-05-14)**: `public/shared/js/dashboard.js:383` ahora pasa `requireCompleteProfile: true` a `ArtistAuth.resolveArtistAuthState`. Antes la unica defensa era `if (!artist.name)` en la linea 449, pero como las filas "stub" (creadas por triggers de auth o por la antigua quick-signup) tienen `name = "Artista"` literal — truthy —, pasaban el check y se mostraba el dashboard con todos los campos en "-". Ahora el resolver llama a `ArtistRegistrationProgress.analyzeArtistProfile`, detecta `isBootstrapName(name) === true`, devuelve `status = 'profile_incomplete'`, y el branch existente en `dashboard.js:401` redirige a `/register-artist` para que el usuario complete los datos faltantes.
- **`buildArtistRegistrationPayload` propaga password (lib/artist-registration.js, 2026-05-14)**: la funcion ahora acepta `options.password` (preferente) o lee `formData.signup_password` (fallback) e incluye `patch.password` solo cuando hay un valor no-vacio. Strings vacios se tratan como ausentes para no sobreescribir un password ya guardado con `''`. Esto activa la persistencia del password durante el wizard pre-auth (cada draft save) y en el finalize endpoint, sin requerir cambios en los callers existentes — `body.data.signup_password` que el wizard ya enviaba ahora fluye automaticamente.
- **`/api/register/artist-finalize` mirrorea password explicitamente (server.js, 2026-05-14)**: pasa `password` via `options.password` a `buildArtistRegistrationPayload` para garantizar que la columna queda escrita junto con el resto del perfil al finalizar el registro, incluso si por algun motivo `formData.signup_password` no esta presente.

El path autenticado usa `savedArtistRow?.password || loadedArtistRecord?.password || formState.data.signup_password || null` en orden de preferencia, para que los usuarios legacy con stub sin password reciban `null` y n8n decida (p. ej. enviar link de reset en lugar de credenciales).
